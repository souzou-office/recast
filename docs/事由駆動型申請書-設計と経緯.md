# recast 新方式 — 事由駆動型 申請書生成（実装プラン）

## ★この変更の性格（最重要・読み違えないこと）★
これは「現行が限界だから直す」修正ではない。「**これが本来あるべき姿だから作る**」。
現行が仮に完璧に動いていても作る。動機は**痛みからの逃避ではなく理想への接近**。
（後述の苦労は「やる理由」ではなく、理想からのズレを示す症状にすぎない。）

---

## 1. 何を作るか（一言）
**「何が起きたか（事由）」をボタンで選ぶと、recast が必要書類を判断し、資料から事実を読んで、
足りない所だけ聞いて、書類一式を完成させる。** ユーザーはフォームを開かない・書類を選ばない。

## 2. 本質と強み（議論の収束点）
- **書類の構造は新しくない。** 穴埋めテンプレで十分（レンダラー不要）。ここで差別化はしない。
- **強みは2つの設定面を"データ"で持つこと**:
  ① 入力側の分岐（事由→どの書類・何を聞く・どの変種か＝木）
  ② 出力テンプレ（穴埋めテンプレ）
  この2つが**コードでなくデータ**なので、エンジン1個で全士業・全ドメインに広がる（横断基盤）。
  縦割りベンダーは知識がコードに焼き込まれてて広がれない。これが構造的な差。
- **入力側の自動化**（散らかった現実＝PDF・画像・Excel を読んで穴を埋める）が、フォーム入力を消す。
- **sweet spot＝行政提出用の定型書類のバッチ**（補助金・許認可・届出・登記・税務…）。
  様式が役所に固定・公開されてる＝「相手の要求仕様」が明示。recast の思想に最も純粋に合致し、
  士業に留まらず一般企業のバックオフィス全体へ広がる。

## 3. アーキテクチャ：今のエンジンは据え置き、前に「事由の木」を足す
```
残す ：テンプレ穴埋めエンジン（出力）= 現行の docx/xlsx マーカー穴埋め一式（正しい・無改修）
足す ：事由ボタン ＋ 分岐の木（どの書類・何を聞く・どの変種かを、事由＋資料から自動決定）
捨てる：AI にテンプレを手術させるパス（analyze の ai モード：組合化・議案削除・行入れ替え）
```
- 今は **人間が手でテンプレフォルダを選ぶ** → 新は **事由ボタン → 木が選ぶ**。
- 「木＝司法書士の段取り判断」をデータ化したもの。単純な事由では小さく、複雑な事由（役員変更）で威力。

### エンジンとデータの分離（"1行も触らない"の意味）
- **エンジン＝コード**（プログラマが1回書く）: ファイルを読む／黄色い穴を見つける／値を入れる／清書する。
- **木＋テンプレ＝データ**（専門家が足す。コードではない）。
- 新しい事由を足す ＝ **木1枚＋テンプレ数枚を置くだけ**。エンジンは無改修。
  （音楽プレイヤー＝エンジン、mp3＝事由データ。曲を足すのにプレイヤーは作り直さない。）
- これが横断性の実体：税理士の書類が欲しければ税理士が木＋テンプレを足す＝同じエンジンで動く。

### 書類の作り方（収束した方針）
- **まるごとテンプレ＋穴埋めが基本**（1事由＝完成書類一式を各1枚、穴を塗るだけ）。
- 大半の申請は **1事由** → 組合せが無いのでテンプレ枚数は爆発しない（事由ごと1セット）。
- **分解（部品化）は当面しない。** 必要になるのは「1書類が"可変の組合せリスト"を抱える所」だけ
  （議事録の議案、契約書の条項）＝複数議案を1総会で、等。これは**後から1箇所だけ**格上げする。
- 将来の一般化モデル（v1 では不要）:「穴＝値 or 小さなテンプレのリスト」の入れ子1個に統一。
  値の穴（会社名）も、テンプレリストの穴（議案）も同じ仕組みの深さ違い。後付けで矛盾しない。
- 構造変種（組合 vs 個人 等）は **AI に書かせず、変種テンプレを丸ごと用意して選ぶ**。

## 4. 使う時の流れ（目的変更の例）
1. サイドバーで会社（例: 株式会社QuantumZero）を選ぶ → recast が顧問先フォルダを読む
2. 事由ボタン `[目的変更] [役員変更] …` から「目的変更」を押す（**最初の入力はこれだけ**）
3. recast が木をたどり、資料で決まる事を埋め、**足りない所だけ聞く**
   （会社名・現目的・株主・代表者は読んで埋め済み → 聞くのは「変更後の目的」「総会開催日」だけ）
4. 答える → 株主総会議事録・株主リストが完成して出る → プレビュー確認 → ダウンロード

## 5. 作る時の流れ（＝事由を足す手順）
新しい事由を足す ＝ **コードを書かず、データを足すだけ**:
1. 木を1枚（YAML）: `事由 → 必要書類 / 聞く分岐 / （複雑なら）変種ルール`
2. テンプレを数枚（docx/xlsx）に黄色マーカーで穴を塗る（既存の穴埋めエンジンがそのまま食う）
3. （事由ボタンは木の一覧から自動生成）

## 6. v1 スコープ（最初の縦1本＝目的変更）
**狙い**: 「事由ボタン → 木 → 既存エンジンで穴埋め」のループを、最も単純な事由で端から端まで通す。
作るもの:
- `data/jirei/mokuteki-henkou.yaml`: 木（必要書類=[議事録, 株主リスト] / 聞く=[変更後の目的, 開催日]）
- テンプレ2枚（`株主総会議事録.docx` / `株主リスト.xlsx`）に黄色マーカーで穴
  - 議事録: 固定文＋黄スロット（会社名・日付・出席状況・現目的・新目的・議長）。slot は事実キーに紐付け
  - スロット→事実: 会社名/議長=登記、株主/議決権=株主名簿、現目的=定款、新目的・日付=聞く
- エンジン側の最小グルー（新規コード。ここは1回だけ書く）:
  - `src/types/jirei.ts` + `src/lib/jirei/loader.ts`（YAML 読込＋型）
  - `src/lib/event-filing/select.ts`（木→必要書類＋質問を出す）/ `ask.ts`（資料で埋まらない分だけ質問）
  - 既存 `src/lib/docx-marker-parser.ts` / `xlsx-marker-parser.ts` / `fill-command-generator.ts` /
    `docx-cleanup.ts` / `xlsx-cleanup.ts` を **import して再利用**（穴埋め・清書は無改修）
  - 事由ボタン UI（まず「目的変更」1個）＋ 既存の生成プレビュー（docx-preview）に流す
- 事実ベース読込は既存の `readAllFilesInFolder` / `file-parsers.ts` / profile を再利用

**ゴール**: テンプレと見分けつかない議事録・株主リストが、ボタン1個＋数問の回答で出る。
通れば 2本目に役員変更（木が変種・必要書類を分岐させる本番）を足す。

## 7. ブランチ / 主なファイル
- ブランチ: 新規 `feat/event-driven-filing`（現行ブランチから分岐。既存システムは無改修で並走）
- 新規: `data/jirei/*.yaml`、`data/jirei-templates/*`（テンプレ）、`src/types/jirei.ts`、
  `src/lib/jirei/loader.ts`、`src/lib/event-filing/{select,ask}.ts`、新 API route、事由ボタン UI
- 再利用（import のみ・無改修）: marker-parser 2種、fill-command-generator、cleanup 2種、
  読込層（readAllFilesInFolder / file-parsers）、verify、docx-preview
- 当面 触らない: 現行の analyze / produce-v2 パス（並走。後で ai モード削除を検討）

## 8. 検証
1. テンプレ2枚を黄マーカーで作成 → docx-preview で素のテンプレ表示確認
2. テスト会社（例: QuantumZero）で「目的変更」ボタン → 事実ベースが定款/登記/株主名簿から埋まるか、
   聞かれるのが「変更後の目的・開催日」の数問だけか
3. 生成 docx/xlsx を docx-preview とレンダリングで確認 → 手作りテンプレと見分けつかないか（忠実度）
4. 既存 verify で原本突合せ
5. 忠実度 OK → 役員変更の木＋変種テンプレを 2本目として実装し同手順で検証

## 付録：ここに至った議論の経緯（保存用）
1. Word/Excel 編集の苦労（officecli 無言失敗・プロセス枯渇・段落ズレ・崩れ・議案ダブり）は全部
   「任意テンプレを自由な AI 出力で外科手術している」一点が根。これは"症状"であって動機ではない。
2. 「AI はざっくり、変換はコード」→ slotId 直接方式（既存）に到達済み。
3. 残る AI 判断は fill/loop/ai のモード分類だけ → 1モード化したい → 議案の構造変更が最後の壁。
4. 最良＝事実×手続きの台帳、書類はビュー…と考えたが、レンダラーは登記の有限な書式では本質的な
   得が無い（テンプレ＋ループ＋変種選択で足りる）と判明。**構造は新しくない**で決着。
5. 強みは構造でなく **入力分岐＋出力テンプレを"データ"で設定できる横断性**。sweet spot は
   行政提出定型書類のバッチ。士業に限らない。ベンダー（縦割り）が構造的に持てない差。
6. 作り方は **まるごとテンプレが基本／組合せが起きる一点（議案）だけ後で部品化**。
7. 結論：**今の穴埋めエンジンは据え置き、前に「事由→書類の木」を足し、AI 手術を捨てる。**
   事由を足す＝木＋テンプレ（データ）を足すだけ＝コード無改修＝横断に効く。
